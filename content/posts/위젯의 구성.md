# 📚 위젯의 구성

## 위젯을 만드는 3가지 방법

### 플러터 기본 내장 위젯

- Flutter에서 기본적인 기능을 제공하고 있는 위젯이다.
- 빠르고 간단하고 Flutter 설치만 하면 바로 사용 가능하다.
- 디자인 커스터마이징에 한계가 있다.
### pub.dev에서 개발자분들이 공유하고 있는 위젯

- 유명 셰프가 만든 특제 양념 소스와 같다.
- 간편하고 기능이 풍부하며 대부분 무료이다.
- 주로 복잡하거나 시간이 오래 걸리는 기능을 쉽게 구현할 때 사용한다.
- 하지만, 정해진 대로 사용을 해야 하고 버전의 충돌 가능성, 보안 및 안정성 확인이 필요하다.
- 사용 방법 예시
  - pubspec.yaml에 추가를 해야함
```yaml
dependencies:
  carousel_slider: ^4.2.1
```

  - import 하여 패키지를 불러와 사용함
```dart
import 'package:carousel_slider/carousel_slider.dart';

CarouselSlider(
  options: CarouselOptions(height: 200.0),
  items: [1, 2, 3, 4, 5].map((i) {
    return Builder(
      builder: (BuildContext context) {
        return Container(
          width: MediaQuery.of(context).size.width,
          margin: EdgeInsets.symmetric(horizontal: 5.0),
          color: Colors.amber,
          child: Text('슬라이드 $i'),
        );
      },
    );
  }).toList(),
)
```

### 본인이 직접 만드는 위젯

- 본인만의 스타일 대로 위젯을 만든다.
- 자주 쓰는 UI 요소를 모듈화 가능하고 여러곳에 재사용이 가능하다.
- 팀 작업 시 코드 정리에 좋다.
- 하지만, 모든 걸 직접 만들어야 해서 시간이 소요되고 처음엔 복잡할 수 있다.
## StatelessWidget과 StatefulWidget

### StatelessWidget

- StatelessWidget은 화면에 나타나고 절대 바뀌지 않는(immutable한 상태) 위젯이다. 
- immutable한 건 안정성, 예측 가능성, 성능 최적화라는 장점이 있다.
- 주로 내부 값이 절대 바뀌지 않는 위젯에 StatelessWidget를 사용한다.
- 이런 건 클릭, 변화해도 바뀔 게 없기 때문에 StatelessWidget을 사용한다.
```dart
Text('환영합니다') // 단순한 글자
Icon(Icons.home)  // 고정된 아이콘
```

- Stateless 사용법
  - 새로운 클래스를 만들어 Flutter의 StatelessWidget을 상속받아 사용한다.
```dart
class FirstMyWidget extends StatelessWidget{
	const FirstMyWidget({super.key});

	@override
  Widget build(BuildContext context) {
    return Text('나의 첫번째 위젯입니다.');
  }
}
```

### StatefulWidget

- StatefulWidget은 화면에 나타나고 동적인 변화를 줄 수 있는 위젯이다. 
- 사용자 입력, 네트워크 응답, 애니메이션, 데이터 변경 등 실시간으로 UI를 바꿔야할 때 필요하다.
- 상태를 기억하고 관리할 수 있으며, setState()함수로 쉽게 업데이트가 가능해 코드가 간결해진다.
- 다만, 상태가 많아지고 자주 바뀌면 성능이 저하될 수 있고 코드가 복잡해지며 유지보수가 어려워진다.
- Stateful 사용법
  - 클래스를 만들어 Flutter의 StatefulWidget을 상속받고, 동적 변화를 만들기 위해 연결된 
State<위젯> 클래스를 상속해 상태를 정의하며, setState() 함수를 호출해 상태를 갱신한다.
  - 필수 구성 요소는 StatefulWidget, createState, State<> + build() 이다.
```dart
class MyWidget extends StatefulWidget {               // 필수 구성요소(Statefulwidget)
  @override                                           
  _MyWidgetState createState() => _MyWidgetState();   // 필수 구성요소(createState())
}                                                     // └── 담당 클래스 확인

class _MyWidgetState extends State<MyWidget> {        // 필수 구성요소(State<>)
  int _counter = 0;                                   // └── 상태 클래스 생성

  void _incrementCounter() {
    setState(() {
      _counter++;
    });
  }

  @override
  Widget build(BuildContext context) {                // 필수 구성요소(build())
    return Scaffold(                                  // └── 상태 클래스의 구성 생성
      appBar: AppBar(
        title: Text('StatefulWidget 예제'),
      ),
      body: Center(
        child: Column(
          mainAxisAlignment: MainAxisAlignment.center,
          children: <Widget>[
            Text('Counting : $_counter'),
            ElevatedButton(
              onPressed: _incrementCounter,
              child: Text('더하기'),
            ),
          ],
        ),
      ),
    );
  }
}
```

# 💡 문제 해결

## 학습하면서 생긴 의문점 및 해답(ChatGPT)

- build() 함수에 대하여 위젯 트리를 반환하는 함수라고 알게됨
**Q1. 위젯을 만드는 다른 클래스는 왜 build 함수를 안쓰는 걸까?**

  A1. 이미 만들어져 있는 위젯(Text, Icon 등)은 내부 자체에 build() 함수를 가지고 있어 
      새로 쓸 필요가 없다.

**Q2. StatelessWidget을 상속할 때 @override로 다시 build 함수를 재정의 하는 이유는?**

  A2. StatelessWidget이 제공하는 build()는 몸체가 비어 있는 함수라 실행 가능한 코드가 없어
      반드시 내가 ‘구체적인 설계도’를 채워 넣어야 실제 위젯을 만들 수 있다.

## 발견된 문제 및 대안

- StatefulWiget에 사용법에서 State<위젯>의 쓰임
**Q1. 클래스는 특성과 기능을 가진 설계도라고 이해했다. 그럼 _MyWidgetState라는 클래스가 안에 
       int _counter = 0 이 있는데 상태를 나타내는 클래스 State<위젯> 을 쓴 이유가 뭘까? **

  A1. State<위젯>을 안쓰면 Flutter는 _MyWidgetState 상태로 인식하지 못하여 setState()를 
      쓰지 못한다. 즉, Flutter가 자동으로 상태를 인식할 수 있도록 하기 위해 사용한다.
