# AI 에이전트 팀 구조, 직접 써보고 전면 재설계했다

## 들어가며

지난 글에서 CLAUDE.md + 서브에이전트 하이브리드 구조로 PM 지속성 문제를 해결했다고 썼다. 이론적으로는 완벽해 보였다. PM이 세션 내내 유지되고, 팀장들이 팀원을 관리하는 3단계 계층 구조였다.

```
Claude Code = PM
├── PlanLeader → PlanMember1, PlanMember2
├── DesignLeader → DesignMember1
└── DevLeader → DevFront, DevBack, DevApp
```

그런데 실제로 돌려보니 작동하지 않았다.

---

## 원인: 에이전트는 에이전트를 호출할 수 없다

### 발견한 문제

테스트 중 이상한 점을 발견했다. PM이 DevLeader를 호출하고, DevLeader가 DevFront를 호출하려 하면 — 아무 일도 일어나지 않았다. DevFront가 실행되지 않는 것이다.

원인을 추적해보니 **Claude Code의 구조적 제약** 때문이었다.

> **서브에이전트로 스폰된 에이전트는 다시 Agent 툴을 사용할 수 없다.**

즉, Claude Code(메인)는 에이전트를 호출할 수 있다. 그런데 그 에이전트는 또 다른 에이전트를 호출할 수 없다. 1단계 중첩은 가능하지만, 2단계 이상은 막혀 있다.

### 왜 이런 제약이 있을까

의도된 설계로 보인다. 이유는 두 가지다.

첫째, **무한 재귀 방지**. 에이전트가 에이전트를 호출하고, 그 에이전트가 또 에이전트를 호출하면 통제 불가능한 루프가 생길 수 있다.

둘째, **권한 범위 제한**. 서브에이전트는 특정 목적에만 집중하도록 설계된다. 오케스트레이션 권한은 최상위 레벨에만 부여하는 것이 안전하다.

### 구체적으로 어디서 막히나

```
Claude Code → PM 스폰:       ✅ 작동
PM → DevLeader 호출 시도:     ❌ PM이 서브에이전트라서 Agent 툴 사용 불가
```

PM.md에 `tools: Agent(PlanLeader, DevLeader, DesignLeader)`라고 명시해도 소용없다. PM이 서브에이전트로 스폰된 순간 그 Agent 툴은 비활성화된다.

결국 기존 3단계 구조는 설계 자체가 작동 불가능한 구조였다.

---

## 의사결정: 어떻게 재설계할 것인가

### 검토한 선택지들

**방안 A: 팀장이 멤버를 흡수**
Leader + Member를 하나로 합쳐 2단계로 줄이는 방법.
- PM → Leader 호출이 작동한다는 전제가 필요했는데, 사실 이것도 불확실했다.

**방안 B: PM이 모든 에이전트 직접 호출**
Leader 계층을 없애고 PM이 DevFront, DevBack 등을 직접 관리.
- 검증자(Verifier), 리뷰어(CodeReviewer) 역할은 어디로 가나?

**방안 C: Claude Code가 PM 역할**
PM 에이전트 자체를 없애고 Claude Code 메인 세션이 모든 에이전트를 직접 호출.
- 이미 CLAUDE.md로 Claude Code를 PM으로 만들고 있었으니, 구조만 바꾸면 된다.

### 선택: 방안 C + 플랫 구조

CLAUDE.md가 이미 "PM.md를 읽고 PM 역할을 해라"고 지시하고 있다. 즉, **Claude Code 자체가 PM**이다. 여기에 서브에이전트를 직접 1단계로 붙이면 된다.

계층을 없애는 대신, **역할 분리**로 품질을 유지한다.

---

## 해결 방안: 플랫 구조 + 역할 특화

### 새 아키텍처

```
Claude Code (= PM, CLAUDE.md → PM.md 참조)
├── [기획]  Planner      — 태스크 세분화 + 접근법 설계
├── [조사]  Researcher   — 시장조사, 경쟁사, 기술분석, 논문
├── [검증]  Verifier     — 자료 신뢰성 확인, 법적 문제 검토
├── [디자인] Designer    — 스타일 시스템, 화면 UI 명세
├── [개발]  DevApp       — Flutter 등 앱 개발
├── [개발]  DevFront     — 웹 프론트엔드 개발
├── [개발]  DevBack      — 백엔드/API/DB 개발
├── [QA]   CodeReviewer  — 코드 품질/보안/아키텍처 리뷰
└── [자문]  Architect    — Opus 기반 고급 판단 (PO 승인 필요)
```

이전 구조와 비교하면 팀장 레이어가 사라지고, 역할이 더 세분화됐다.

### 기존 팀장이 하던 일은 어디로 갔나

팀장의 핵심 역할은 두 가지였다.

1. **태스크 세분화** → **Planner**가 전담한다
2. **2차 검증 (코드리뷰, 자료검증)** → **CodeReviewer**, **Verifier**가 전담한다

오히려 역할을 더 명확하게 분리했다. 팀장은 세분화도 하고 검증도 하고 보고도 하는 복합 역할이었는데, 이제 각각 전문 에이전트가 담당한다.

## 설계 포인트: 토큰 효율

### 서브에이전트는 프로젝트 히스토리를 모른다

가장 중요한 설계 원칙은 이것이다.

> **서브에이전트는 컨텍스트가 없다. PM이 호출 시 필요한 모든 정보를 직접 구성해서 전달한다.**

기존 구조에서는 팀장이 팀원에게 컨텍스트를 전달했다. 이제 PM이 그 역할을 직접 한다. 매번 호출할 때 기술스택, 작업 범위, 파일 경로, 접근 방법을 포함해서 전달한다.

```
DevBack 호출 시 PM이 전달하는 것:
- 서브태스크 설명
- 기술스택 (예: NestJS + PostgreSQL)
- 수정할 파일 경로
- 스코프 범위 (어떤 파일은 건드리면 안 되는지)
- 접근 방법 (어떻게 구현할지 방향)
```

이렇게 하면 서브에이전트가 "어디서 시작해야 하지?"를 고민하는 시간이 없다. 바로 실행에 들어간다.

### QA는 자동으로

QA를 별도 단계로 두면 잊어버리거나 생략하게 된다. 그래서 PM.md에 규칙으로 못 박았다.

```
규칙 1: Researcher 완료 → 즉시 Verifier 호출 (생략 불가)
규칙 2: 개발자 완료 → 즉시 CodeReviewer 호출 (생략 불가)
규칙 3: 문제 발견 → 원래 에이전트에 수정 요청 → 재검증
규칙 4: 3회 실패 → PM이 PO에게 즉시 보고
```

팀장이 없어도 품질이 유지되는 이유다.

## 설계 포인트: 각 에이전트의 역할

### Planner — 태스크를 실행 가능하게 쪼갠다

PM이 "로그인 기능 만들어"라는 지시를 받으면 Planner를 먼저 호출한다. Planner는 이것을 구체적인 서브태스크로 쪼개고, 각 태스크의 접근 방법까지 정의해서 돌려준다.

```
태스크: 로그인 기능 구현

서브태스크:
1. [DevBack] JWT 인증 서비스 구현 (High)
   접근법: auth.service.ts 신규 생성, bcrypt 비밀번호 해시, 24시간 만료 토큰
2. [DevFront] 로그인 화면 구현 (High, #1 완료 후)
   접근법: login_page.tsx, 기존 color/typography 토큰 사용
3. [CodeReviewer] 코드 리뷰 (Medium, #1, #2 완료 후)
   접근법: 보안(auth bypass, hardcoded secret), IBA 준수 집중 검토
```

PM이 이걸 승인하고 순서대로 에이전트를 호출하면 된다.

### map 파일 — 에이전트의 내비게이션

각 개발 에이전트는 첫 호출 시 `app-map.md`, `front-map.md`, `back-map.md`를 생성하고, 작업할 때마다 업데이트한다.

이 파일의 목적은 단 하나다.

> **"이 파일 하나만 읽으면 수정해야 할 파일 위치를 바로 찾는다."**

디렉토리를 탐색할 필요가 없다. 파일명과 역할이 전부 여기 있다. 에이전트가 작업 후 업데이트하지 않으면 다음 에이전트가 잘못된 위치를 참조할 수 있어서, 업데이트를 보고 형식에 강제로 포함시켰다.

### Architect — 비용이 있는 전문가

Architect는 Opus 모델을 사용한다. 가장 똑똑하지만 가장 비싸다. 그래서 아무 때나 호출하지 않는다.

PM이 판단하기 어려운 상황 — 아키텍처 트레이드오프, 보안 전문 판단, 법적 리스크 등 — 에서만 PO에게 먼저 승인을 받고 호출한다. Architect는 조언만 한다. 코드를 쓰거나 파일을 수정하지 않는다.

## 결과

### 달라진 것

| 항목 | 기존 | 변경 후 |
|------|------|---------|
| 구조 | 3단계 계층 (PM → 팀장 → 팀원) | 2단계 플랫 (PM → 에이전트) |
| 팀장 역할 | 세분화 + 검증 + 보고 | Planner + CodeReviewer/Verifier로 분리 |
| QA | 팀장 판단에 의존 | PM.md에 자동화 규칙으로 강제 |
| 토큰 효율 | 팀장도 Sonnet, 오버헤드 큼 | 조사/검증은 Haiku, 개발은 Sonnet |
| 컨텍스트 전달 | 팀장 → 팀원 계층적 전달 | PM이 직접 필요한 정보만 구성해서 전달 |

### 남아있는 한계

세션이 끊기면 여전히 컨텍스트가 초기화된다. `claude --resume`으로 이전 세션을 이어갈 수 있지만, 긴 작업 중 세션이 끊기면 PM이 어디까지 진행됐는지 다시 파악해야 한다.

이 문제는 `meetings/pm-inbox.md`와 각 팀의 로그 파일로 어느 정도 완화된다. 에이전트들이 작업 완료 후 로그를 남기기 때문에, PM이 다음 세션에서 로그를 읽으면 현황을 파악할 수 있다. 완벽하진 않지만 실용적으로 동작한다.

## 마치며

처음 설계할 때는 "실제 팀처럼 계층 구조를 만들자"는 생각이 강했다. 그게 더 체계적으로 보였으니까. 그런데 실제로 돌려보니 그 계층 구조가 작동하지 않았다.

결국 제약 안에서 다시 설계했고, 오히려 더 단순하고 토큰 효율도 좋은 구조가 나왔다.

AI 에이전트 시스템을 설계할 때 한 가지 교훈을 얻었다.

> **"얼마나 복잡하게 설계할 수 있느냐"보다 "실제로 작동하는 가장 단순한 구조가 무엇인가"를 먼저 찾아야 한다.**

설계는 이론이고, 작동은 현실이다.
